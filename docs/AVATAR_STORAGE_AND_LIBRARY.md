# Avatar storage and library

## Current implementation (library feature)

The **library feature is implemented end-to-end**:

- **Backend (Vercel)**  
  - **Avatar API** is deployed on Vercel (`avatar-creator-server`).  
  - **Storage:** **Vercel KV** (Redis-backed). Keys: `avatar:{id}`, `owner:{ownerId}` (list of avatar IDs), `library:ids` (list of published avatar IDs).  
  - **APIs:** POST/GET `/api/avatars`, GET/PATCH/DELETE `/api/avatars/:id`, GET `/api/library` (published avatars only).

- **Fly**  
  - Used for the **model-generation worker** (Python job that produces the GLB). Avatar records and library list live on Vercel + Vercel KV, not on Fly.

- **Client**  
  - **Avatar tab:** “My avatars” (list by `ownerId`), Use / Publish / Delete. Publish sets `publishedAt` via PATCH.  
  - **Library tab:** GET `/api/library`, grid of published avatars; “Use this avatar” sets current avatar (persisted in sessionStorage and RUN `appStorage` so it survives refresh).  
  - **Persistence:** Selected avatar is written to both sessionStorage and `RundotGameAPI.appStorage` when RUN SDK is available, and hydrated from app storage on load.

---

## Goals (original plan)

1. **Persist the preview image** and link it to the GLB so we can show thumbnails and use them in a library.
2. **Support “publish to library”** later: users can publish an avatar; others can browse (preview image) and use (GLB).

---

## Current state

- **Image**: Generated by `/api/v1/image`; may be returned as base64 and/or uploaded to Vercel Blob (`v1/preview-xxx.png`). When starting the 3D job we send either `image` (base64) or `image_url` to `/api/v1/model`. If base64, the API uploads to Blob as `jobs/{jobId}/source.png` and gets a URL. That URL is **not** stored in the job record—only passed to the worker.
- **GLB**: Worker uploads to Blob (`jobs/{jobId}/rigged.glb` etc.) and updates the job via `kv_update_fn` with `result: { glb_url, fbx_url, ... }`.
- **Client**: Persists only `glbUrl` in sessionStorage (`echoAvatar_glbUrl`). No preview URL, no “avatar” entity.

So today the image is often already in Blob (either from v1/image or from `jobs/{jobId}/source.png`), but we don’t store its URL on the job or in the app, so we can’t reliably show “preview + GLB” as one unit or use it for a library.

## Phase 1: Link preview image + GLB (no new storage)

**Idea:** Store the source/preview image URL on the **job** when we create it. The image is already in Blob; we only need to record the URL and return it with the job.

### Backend (avatar-creator-server)

1. **Persist `previewImageUrl` on the job**
   - In `api/v1/model/index.js`, when creating the job (both base64 and `image_url` paths), call:
     - `setJob(jobId, { status: 'queued', createdAt: Date.now(), previewImageUrl: imageUrl })`
   - `imageUrl` is either:
     - The blob URL we get after uploading base64 to `jobs/${jobId}/source.png`, or
     - The client-supplied `image_url` (e.g. from `/api/v1/image` blob upload).
   - No new storage: we only add a field to the existing job KV value.

2. **Optional: add to `result` when job completes**
   - Worker could pass back the same URL in `result.preview_image_url`, or the client can keep using `job.previewImageUrl` from the initial job payload. Prefer keeping it at top-level `job.previewImageUrl` so a single GET `/api/job/:id` returns both:
     - `job.previewImageUrl` – preview image (stable Blob URL)
     - `job.result.glb_url` – GLB when status is `done`

3. **GET `/api/job/:id`**
   - Already returns the full job object; once we add `previewImageUrl` to the job, the response automatically includes it. No change needed in the job GET handler.

### Client (echoAvatar)

1. **When the model job completes**
   - From the last poll response we have `job.result.glb_url` and `job.previewImageUrl`.
   - Persist both:
     - **Option A (minimal):** Keep using sessionStorage for `glbUrl` only; add a second key (e.g. `echoAvatar_previewImageUrl`) and store the preview URL there. Context/schema: “current avatar” = `glbUrl` + optional `previewImageUrl”.
     - **Option B (single object):** Store one object, e.g. `{ glbUrl, previewImageUrl }` in sessionStorage (and optionally in RUN.game `appStorage` for cross-session). One key, one JSON value.

2. **AvatarGlbContext**
   - Extend to hold `previewImageUrl` (or a single `avatar` object `{ glbUrl, previewImageUrl }`). Load/save it from the chosen storage so the Avatar tab (and future library cards) can show the preview without refetching the job.

3. **Avatar tab**
   - In “view” mode, you can keep showing the 3D preview; if we add a list of “my avatars” later, we’d use `previewImageUrl` for list/thumbnails.

**Deliverables for Phase 1**

- Backend: `previewImageUrl` on job in `api/v1/model` (both code paths).
- Client: Persist and expose `previewImageUrl` alongside `glbUrl` (storage + context), and use it where a preview is needed (e.g. future “my avatars” or library cards).

## Phase 2: Avatar entity and “publish to library”

**Idea:** Introduce an **avatar** entity (preview + GLB + metadata) and a **library** of published avatars. Other users can browse by preview and use the GLB.

### Data model

- **Avatar record** (stored in KV or a future DB):
  - `id` – stable avatar id (e.g. UUID or `avatar_${timestamp}_${random}`).
  - `previewImageUrl` – Blob URL for the preview image (or a copy under a stable path; see below).
  - `glbUrl` – Blob URL for the GLB.
  - `jobId` – optional; link back to the job that created it.
  - `ownerId` – RUN.game profile id (or similar) for “my avatars” and moderation.
  - `name` – optional display name.
  - `createdAt`, `publishedAt` – `publishedAt` set when user “publishes to library”.

- **Stability:** Today, GLB and image live under `jobs/{jobId}/...`. If you later delete old jobs, those URLs break. For library avatars you can:
  - **Option A:** Copy assets to a long-lived path when publishing, e.g. `avatars/{avatarId}/preview.png` and `avatars/{avatarId}/model.glb`, and store those URLs in the avatar record. Job blobs can still be cleaned up.
  - **Option B:** Keep using job URLs and avoid deleting jobs for published avatars (simpler but ties lifecycle to jobs).

### Backend

1. **Storage**
   - Use KV (e.g. `avatar:{id}`) or a small DB table. Schema: fields above. List avatars by `ownerId` for “my avatars”, and by `publishedAt != null` for “library” (with optional pagination).

2. **APIs**
   - **POST /api/avatars** (or /api/v1/avatars)  
     - Body: `{ jobId }` or `{ previewImageUrl, glbUrl }` (+ optional `name`).  
     - Creates an avatar record (and optionally copies preview + GLB to `avatars/{id}/...`).  
     - Returns `{ id, previewImageUrl, glbUrl, ... }`.
   - **GET /api/avatars** – list current user’s avatars (needs auth/profile id).
   - **GET /api/avatars/:id** – get one avatar (for “use this avatar”).
   - **PATCH /api/avatars/:id** – e.g. set `name`, or set `publishedAt` for “publish to library”.
   - **GET /api/library** (or GET /api/avatars?published=true) – list published avatars (preview + glb + owner/name); used for browse.

3. **Publish flow**
   - User selects “Publish to library” → PATCH `/api/avatars/:id` with `publishedAt: Date.now()` (and maybe copy assets to `avatars/{id}/...` if using Option A). Library list is GET /api/library.

### Client

1. **After generating a new avatar (Phase 1 done)**
   - Call POST /api/avatars with `{ jobId }` (or `{ previewImageUrl, glbUrl }`) so the backend creates an avatar record. Optionally set that as “current avatar” (same as today’s setGlbUrl + setPreviewImageUrl).

2. **“My avatars”**
   - GET /api/avatars; show list of cards (preview image + name); “Use” sets glbUrl (and preview) as current; “Publish” calls PATCH to set publishedAt.

3. **Library tab / screen**
   - GET /api/library; show grid of published avatars (preview + name + owner). “Use this avatar” sets that entry’s glbUrl (and preview) as current and persists in app storage.

4. **Linking**
   - Everywhere we persist “current avatar,” store both `glbUrl` and `previewImageUrl` (and optionally `avatarId`). Library entries are already linked: one record = one preview + one GLB.

---

## Summary

| What | Phase 1 | Phase 2 |
|------|--------|--------|
| **Preview stored?** | Yes – already in Blob; we store URL on the **job** as `previewImageUrl`. | Same; avatar record also holds `previewImageUrl` (and optionally a copy under `avatars/{id}/`). |
| **Linked to GLB?** | Yes – same job: `job.previewImageUrl` + `job.result.glb_url`. | Yes – one avatar record: `previewImageUrl` + `glbUrl`. |
| **Client persistence** | Persist both `glbUrl` and `previewImageUrl` (sessionStorage or appStorage). | Plus “my avatars” and “library” from API; current avatar still glbUrl + previewImageUrl. |
| **Library** | N/A | Avatar entity + publish + GET /api/library; other users see previews and can use GLB. |

Phase 1 gives you stored preview + link to GLB with minimal backend change (one extra field on the job). Phase 2 adds the avatar entity and publish-to-library flow so preview and GLB stay linked and shareable.
